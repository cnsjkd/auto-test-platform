## 1. 调研依据

本阶段重点核验能力边界，而不是泛泛判断“哪个好”。参考方向包括：

1. Android ADB：client-server 架构、`adb devices -l`、`adb -s <serial>`、`adb shell`、APK 安装、文件 push/pull、截图、录屏、USB debugging/RSA 授权。
2. Android logcat：日志 buffer、过滤规则、输出格式、日志等级与限制。
3. Android dumpsys：`gfxinfo/meminfo/procstats/batterystats/netstats/input` 等诊断能力。
4. Appium 2 + UiAutomator2 Driver：Android native/hybrid/mobile web 自动化，依赖 SDK/JDK/USB debugging，支持 capabilities 和多设备并发。
5. Python uiautomator2/openatx：截图、点击、滑动、dump hierarchy、元素定位、应用管理、设备信息。
6. DeviceFarmer/STF：浏览器远程设备农场，能力强但依赖重，Windows 安装不友好。
7. pytest / pytest-xdist：fixture、参数化、插件生态、多 worker 并行，需要平台自己做设备分配锁。
8. Allure Pytest：结果目录、附件、步骤、元数据、HTML 报告生成。
9. Prometheus / Grafana：性能、稳定性、趋势与告警。
10. Android MediaRecorder / AudioRecord：音频录制链路可行，但 ADB 本身不能完整、精确验证会议音频质量。

## 2. 技术选型对比矩阵

| 方案 | 学习/落地成本 | 生态成熟度 | Windows 本机部署 | UI 自动化能力 | ADB/设备命令能力 | 多设备并发 | 企业化扩展 | 结论 |
|---|---:|---:|---:|---:|---:|---:|---:|---|
| Appium 2 + UiAutomator2 + pytest | 3 | 5 | 3 | 5 | 4 | 4 | 5 | 行业标准强，但起步依赖较多 |
| Python uiautomator2/openatx + adbutils/pytest | 5 | 4 | 5 | 4 | 5 | 4 | 4 | 最适合先在 Windows 真机跑通，再封装平台 |
| 纯 ADB + pytest | 5 | 5 | 5 | 2 | 5 | 4 | 3 | 稳定底座，但复杂 UI 选择器弱 |
| DeviceFarmer/STF/OpenSTF | 2 | 4 | 1 | 4 | 4 | 5 | 5 | 适合后期 Linux 设备农场，不适合 Windows MVP |
| Appium + Selenium Grid | 2 | 5 | 2 | 5 | 3 | 5 | 5 | 只解决 WebDriver 路由，不解决设备资源分配 |
| Allure + Prometheus/Grafana | 4 | 5 | 4 | N/A | N/A | N/A | 5 | 应作为平台配套层，不是执行引擎 |

### 选型判断

1. 不建议纯 Appium 起步：Appium 是企业标准，但初始依赖 Node/JDK/Android SDK/Appium server/driver/capabilities/security flags，先跑通基础操作会慢。
2. 不建议纯 ADB 作为唯一方案：纯 ADB 对截图、录屏、logcat、dumpsys、点击滑动很强，但 UI 层级、元素等待、选择器、断言表达弱。
3. 不建议 STF 作为 MVP 核心：STF 是设备农场，不是测试框架；依赖重，Windows 不友好。
4. 推荐“ADB 底座 + uiautomator2 主驱动 + pytest 执行 + Allure 报告 + FastAPI/React 控制台 + PostgreSQL 元数据”。Appium 2 作为后续兼容适配器加入，不阻塞 MVP。

## 3. 推荐最终稳定组合

### 执行底座

- Android SDK Platform Tools / ADB：设备发现、截图、录屏、logcat、dumpsys、文件传输、shell 命令、安装卸载。
- Python 3.11/3.12：统一测试平台语言。
- pytest：测试用例执行、fixture、参数化、失败断言、插件扩展。
- pytest-xdist：多进程并发执行，设备分配必须由平台 scheduler/lock 管理。

### UI 自动化主通道

- Python uiautomator2/openatx：点击、滑动、下拉、元素定位、dump hierarchy、截图、Toast、应用启动/停止。
- ADB 系统动作与像素兜底通道：`adb shell input keyevent`、通知栏/快捷设置展开、`screencap`、`screenrecord`、`uiautomator dump`。业务元素默认不得使用像素坐标；若控件确实无法语义定位且不能跳过自动化，可用 `adb shell input tap/swipe` 做“像素坐标兜底”，但必须强制记录原因、坐标、分辨率/方向、截图证据和改造建议，并在报告中标红展示。

### Appium 兼容通道（Phase 2/3）

- Appium 2 + UiAutomator2 driver：用于 W3C WebDriver 标准、跨语言脚本、WebView/Chrome、企业标准接入。
- 每台设备独立 `udid/systemPort/chromedriverPort/mjpegServerPort`。

### 报告与看板

- Allure Pytest：每次测试运行的用例级报告，挂截图、录屏、logcat、UI XML、音频文件、性能采样 CSV/JSON。
- PostgreSQL：平台元数据、测试计划、运行记录、设备、结果、artifact 索引。
- 文件存储：本地目录起步，后续切 NAS/MinIO；DB 只存路径和元数据。
- Prometheus + Grafana：稳定性/性能长周期趋势，不替代 Allure。

### Web 平台

- 后端：FastAPI + SQLAlchemy + Alembic。
- 前端：React + Ant Design。
- 执行 worker：Python subprocess/process pool，按设备锁调度 pytest 子进程。
- 认证：本机 MVP 可关闭登录；企业版启用 JWT + RBAC。

## 4. 核心功能可行性验证

| 功能 | 可行性 | 推荐实现 | 风险/边界 |
|---|---|---|---|
| Windows 上发现真机 | 可行 | `adb devices -l`，识别 serial/state | 驱动、线缆、USB hub、设备未授权会失败 |
| 多设备指定 | 可行 | 所有 ADB 命令强制 `adb -s <serial>` | 未指定 serial 会出现 more than one device/emulator |
| 截图 | 可行 | `adb exec-out screencap -p` 或 uiautomator2 screenshot | 安全窗口/DRM 可能黑屏 |
| 点击/滑动/下拉 | 可行 | uiautomator2 基于 resource-id/text/description/accessibility label/hierarchy path 执行；ADB 用于 Home/Back/通知栏展开等系统级动作；像素坐标仅作为显式兜底策略 | 系统可能禁用通知栏；自绘控件需可测试性改造或图像识别专项。若短期必须像素坐标兜底，报告必须标注原因、风险、坐标、分辨率/方向 |
| UI 层级 dump | 可行 | uiautomator2 `dump_hierarchy()`；ADB `uiautomator dump` | 自绘控件、WebView、Surface 信息不足 |
| 录屏 | 可行 | `adb shell screenrecord /sdcard/x.mp4` + pull | 安全内容可能录不到；长录制需分段；音频不一定包含 |
| logcat | 可行 | `adb logcat -b main,system,crash -v threadtime` | 日志是环形 buffer；部分日志需权限/root |
| 性能采集 | 可行 | `dumpsys gfxinfo/meminfo/procstats/batterystats/netstats/input`，加 `top`/`pidof` | 指标设备差异大 |
| 电量/温度 | 部分可行 | `dumpsys battery`、`batterystats`、`thermalservice` | 温度/thermal 节点可能缺失或需权限 |
| 音频播放验证 | 可行 | push 标准音频文件，启动系统播放器或 companion APK 播放 | 系统播放器差异大 |
| 音频录音验证 | 部分可行 | companion APK 用 MediaRecorder/AudioRecord 录音后 pull 文件分析 | ADB 本身不能可靠获取麦克风 PCM |
| 会议音频链路质量 | 高成本可行 | 外部参考音源/声卡/麦克风 + companion APK + 算法分析 | 不能只靠 ADB 声称全链路质量已验证 |
| 多设备并发 | 可行 | 设备分配锁 + 每设备 worker + pytest 参数化/xdist | USB 带宽、电源、ADB server 稳定性是瓶颈 |
| 测试报告 | 可行 | Allure 附件 + 平台 DB 摘要 | Allure 结果目录要按 run 隔离 |
| Web 控制台 | 可行 | FastAPI + React AntD + WebSocket | 不应在 MVP 一开始做大而全权限系统 |
| 远程设备农场 | 后期可行 | Linux 主机 + DeviceFarmer/STF 或自研 device-agent | Windows 本机不适合上 STF |

## 6. 服务边界

1. **Device Service**：设备发现、状态、占用、能力探测，不执行测试逻辑。
2. **Command Service**：封装单次设备命令，如语义化点击/输入/滑动、系统 keyevent、screenshot/logcat/shell；像素坐标接口允许存在，但只能通过 `pixel_fallback=true` 的显式参数调用，并必须填写 fallback_reason、x/y、screen_width/screen_height、orientation、risk_note。
3. **Run/Scheduler Service**：任务排队、设备锁、worker 派发、状态机。
4. **Pytest Runner**：执行测试，通过结果适配器回写。
5. **Artifact Service**：统一管理截图、录屏、日志、音频、UI XML、Allure 文件。
6. **Metrics Service**：采集并规范化 dumpsys/top 等性能数据。
7. **Audio Check Service**：管理音频样本、播放/录音动作、音频分析结果。
8. **Report Service**：生成 Allure 报告并聚合平台摘要。

## 8. 数据库设计初稿

核心表：

| 表名 | 说明 | 核心索引 |
|---|---|---|
| users | 用户与角色 | username unique, role |
| devices | ADB 设备资产 | serial unique, status, last_seen_at, model |
| device_sessions | 设备锁/占用会话 | device_id, run_id, status, expires_at |
| test_cases | 测试用例 | type, priority, enabled |
| test_suites | 测试套件 | name, deleted_at |
| suite_cases | 套件与用例关系 | suite_id, case_id, unique(suite_id, case_id) |
| test_plans | 测试计划 | suite_id, enabled |
| test_runs | 测试运行 | status, plan_id, suite_id, triggered_by, started_at |
| test_run_devices | 运行与设备关系 | run_id, device_id, status |
| test_results | 用例结果 | run_id, case_id, device_id, status |
| artifacts | 截图/视频/logcat/UI XML/Allure 等附件 | run_id, result_id, device_id, type, created_at |
| metrics | 性能/稳定性采样 | run_id, device_id, metric_name, sampled_at |
| device_commands | 设备命令审计 | device_id, run_id, command_type, status, started_at |
| audio_checks | 音频链路检查 | run_id, device_id, status |

迁移策略：

1. 使用 Alembic 管理 DB migration。
2. MVP 可用 PostgreSQL 本机服务；如未安装 DB，可提供 SQLite dev mode，但企业模式以 PostgreSQL 为准。
3. artifact 文件目录按 `artifacts/{run_id}/{device_serial}/...` 分层；DB 仅保存路径、类型、大小、checksum。
4. 设备发现采用 upsert：以 `serial` 为唯一标识更新设备属性。
5. 删除策略：业务数据软删除 `deleted_at`；大文件按保留策略清理。

## 9. Windows 本机部署与运行方式

必备：

1. Android SDK Platform Tools，`adb` 加入 PATH。
2. Python 3.11/3.12。
3. A2 真机开启 Developer options + USB debugging，并授权 RSA。
4. PostgreSQL 本机服务或 Docker/PostgreSQL；短期可 SQLite dev mode。
5. 可选：JDK + Node + Appium 2，Phase 2 再安装。

启动形态：

```text
start-api.bat      -> FastAPI 后端
start-worker.bat   -> pytest/device worker
start-web.bat      -> React 前端
adb devices -l     -> 平台启动前健康检查
```

## 10. 迭代路线

### Phase 0：真机基础闭环

- `adb devices -l`
- 获取 getprop、屏幕尺寸、系统版本
- 截图
- 下拉通知栏/快捷设置
- 点击/滑动/返回/Home
- dump hierarchy
- screenrecord
- logcat 分片保存
- dumpsys meminfo/battery/gfxinfo 基础采样

### Phase 1：pytest + Allure 自动化闭环

- 封装 device fixture。
- 产出第一批 smoke 用例。
- 失败自动截图、dump、logcat。
- Allure 报告可打开。

### Phase 2：Web 控制台与 DB

- 设备列表、手动命令、用例列表、运行测试、查看报告。
- PostgreSQL 入库 run/result/artifact/device。

### Phase 3：音频/性能/稳定性专题

- 长时间运行、重启恢复、内存趋势、温度/电量趋势。
- companion APK 做统一播放/录音。
- 音频文件基础分析。

### Phase 4：企业化扩展

- Appium adapter。
- Prometheus/Grafana。
- RBAC/审计。
- 多设备/多主机 agent。
- 必要时引入 DeviceFarmer/STF 做远程人工设备控制。

## 11. 风险与不可行/高成本警告

1. “能包含的全部包含，能测试的全部测试”不能理解为无限能力；无 root、无系统签名、无厂商私有接口时，部分底层指标、受保护日志、安全窗口、内部音频流不可完全采集。
2. 音频链路是最高风险点：ADB 可以触发播放、拉取文件、跑 companion APK，但不能单独保证会议麦克风/扬声器/降噪/回声消除/阵列算法的端到端质量。
3. UI 层级并非总可靠：自绘控件、SurfaceView、WebView、系统安全页面可能 dump 不出可用元素；处理方式优先是推动 accessibility id/test id 等可测试性改造、提供测试入口或进入图像识别专项。若短期只能靠像素坐标触达，不能放弃自动化测试，但必须以“像素坐标兜底”执行并显式告警。
4. 像素坐标兜底不是默认策略：必须记录 fallback_reason、x/y、screen_width/screen_height、orientation、前后截图、稳定性风险、后续可测试性改造建议；报告与用例详情必须标红展示，便于用户知情决策。
5. Appium 版本与 Android 版本要自动探测。
6. STF 不适合 Windows MVP。
7. Selenium Grid 不能替代设备管理。
8. 多设备并发受 USB hub 供电、线缆质量、ADB server 稳定性影响。
9. logcat/dumpsys 输出随 ROM 变化，解析器必须容错，原始日志要保存。
10. Allure 结果目录必须每次 run 隔离，避免历史结果污染。

## 12. 共享内存池写入

### 技术约束

1. 当前项目以 Windows 本机 + ADB 真机为起点；所有设备操作必须基于 `adb -s <serial>`，不可假设单设备。
2. A2 设备系统版本、API level、屏幕尺寸、厂商 ROM 行为由平台启动时自动探测；不在设计阶段硬编码。
3. ADB/USB debugging/RSA 授权是所有能力前置条件；未授权设备只能展示状态，不能执行测试。
4. 无 root/无系统签名场景下，部分底层指标、受保护日志、安全窗口、内部音频流不可完全采集。
5. UI 自动化必须语义定位优先：resource-id、text、description、accessibility label、class + hierarchy path、稳定语义锚点。像素坐标仅允许作为显式兜底能力，不能静默替代语义定位；图像识别作为专项能力，使用时也必须显式标注。
6. 音频链路高可信验证必须引入 companion APK 或外部音频闭环设备；纯 ADB 只能做有限验证。
7. 多设备并发必须有设备锁、端口隔离、artifact 隔离、run 目录隔离。
8. Allure 每次运行必须独立结果目录。

### 选型结论

1. 推荐核心组合：ADB 底座 + Python uiautomator2 主 UI 驱动 + pytest 执行 + Allure 报告 + FastAPI 后端 + React/Ant Design 前端 + PostgreSQL 元数据 + 文件 artifact 存储。
2. 纯 ADB 作为设备命令、性能采集、系统级 keyevent 和像素坐标兜底通道；像素坐标不能替代默认语义定位，只能在控件无法语义定位且必须自动化时显式启用，并强制进入报告风险说明。
3. Appium 2 + UiAutomator2 作为 Phase 2/3 企业兼容 adapter，不阻塞 MVP 首轮真机跑通。
4. Prometheus/Grafana 用于长期性能和稳定性趋势，不替代 Allure 用例报告。
5. DeviceFarmer/STF 只在后期 Linux 多设备实验室阶段评估，不进入 Windows 本机 MVP 核心路径。

### 不可行/高成本警告

1. “全部测试”不能覆盖 Android 安全边界之外的内容。
2. 会议音频端到端质量测试是专项工程，不能承诺仅靠 ADB 完成。
3. Selenium Grid 不能替代设备管理。
4. Windows 上直接部署 STF/OpenSTF 成本高且官方不友好。
5. 多设备规模化的主要瓶颈不是代码，而是 USB 供电、线缆、Hub、ADB server、设备散热与长期稳定性。
