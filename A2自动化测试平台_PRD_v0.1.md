## 1. 问题陈述

### 谁在什么场景下遇到了什么痛点？

- **测试/交付团队**：需要在当前 Windows 电脑上接入一台真实 A2 Android 设备，从 0 开始验证 UI 操作、设备状态、音频链路、稳定性与性能，但没有现成自动化资产、没有设备池、没有报告系统。
- **研发/架构团队**：需要快速复现真机问题，拿到设备日志、截图、录屏、性能数据，而不是依赖测试人员口述。
- **项目/产品负责人**：需要知道每次构建/固件/应用迭代是否可交付，尤其是会议办公场景中的录音、转写、下拉设置、网络、长时间运行等核心路径是否稳定。

### 现在怎么解决？为什么不行？

当前常见方式是：人工点测 + ADB 命令 + 零散脚本 + Appium/uiautomator2/pytest/Allure 等框架拼装。

问题：

1. Windows 本机从 0 搭环境门槛高，ADB 驱动、USB 授权、SDK、设备识别、权限弹窗都会阻断第一步。
2. 开源框架只解决“执行”，不解决“企业平台”：设备接入、用例资产、失败证据、报告趋势、设备池、音频链路、长期稳定性都要二次产品化。
3. 云测平台覆盖大量通用手机，但难覆盖 A2 这种专用会议办公硬件、定制 ROM、音频外设、企业内网、线下 USB 调试场景。
4. 用户说“全部测试”本质是风险诉求：希望任何可自动化验证的质量风险都能逐步纳入，而不是第一版就做一个臃肿平台。

## 2. 目标用户

### 主要用户画像：测试负责人 / 自动化测试工程师

- 经验：3 年以上移动端或硬件测试经验。
- 场景：拿到 A2 真机后，需要快速完成冒烟、回归、稳定性和交付验证。
- 技术水平：会用 ADB、看日志、理解测试用例，但不希望每次都手写脚本和维护零散报告。
- 核心诉求：一键识别设备、快速编排基础 UI 操作、稳定执行、失败有证据、报告可交付。

### 次要用户画像

- **Android/设备研发工程师**：通过平台复现 UI/系统/音频/性能问题，获取 logcat、截图、录屏、设备状态、性能曲线。
- **项目经理/产品经理/交付负责人**：关注当前版本是否通过、哪些模块高风险、失败是否影响交付。
- **未来设备池管理员/测试平台管理员**：当设备从 1 台扩到多台时，统一设备状态、占用、任务队列、结果归档和权限。

## 3. 核心场景

1. **首次接入真机**：在当前 Windows 电脑连接 A2，平台自动识别 ADB 状态、设备型号、Android 版本、分辨率、电量、存储、网络、应用列表、权限状态。
2. **基础 UI 冒烟**：先跑通下拉框、点击、输入、返回、断言、截图等基础操作，证明平台能真实控制 A2。
3. **回归测试**：将会议办公核心路径沉淀为用例集，例如设置项、录音入口、会议创建、转写入口、网络切换、账号/同步状态等。
4. **设备命令与接口验证**：统一执行 ADB shell、设备信息采集、应用安装/卸载/启动/清数据、系统状态采集。
5. **音频链路验证**：逐步覆盖输入输出、延迟、静音、噪声、长录音、异常中断等场景。
6. **稳定性与性能**：长时间会议、待机唤醒、充电/低电量、网络波动、反复启动、内存/CPU/电量趋势。
7. **多设备并发与兼容性**：未来从单机单设备扩展到本地设备池，支持多台 A2 或不同系统版本并发执行。
8. **报告看板**：每次执行生成可审计报告，给测试、研发、项目负责人各自可用的信息视图。

## 5. 市场空白与差异化定位

### 云测平台共同空白

1. 云端有通用设备，但没有用户手里的 A2 专用硬件。
2. 从 0 接入不够友好，通常假设已有 App 包、测试脚本和 CI。
3. 音频链路不是普通 App UI 自动化能覆盖的。
4. 本地 Windows 单机到企业设备池缺少渐进路径。

### 开源自建方案共同空白

1. 执行框架很多，但“设备接入—用例编排—执行调度—证据采集—报告看板—资产沉淀”没有一体化产品体验。
2. Windows 环境下 ADB、SDK、驱动、USB 授权、权限弹窗是第一天主要阻碍。
3. Allure 能展示结果，但不能天然表达设备状态、ADB 日志、音频指标、长稳趋势、多设备占用。

### 定位一句话

面向专用 Android 会议办公硬件的 Local-first 真机自动化测试平台，从当前 Windows 单机 + ADB 真机 0 资产起步，先跑通基础 UI 操作，再平滑扩展到企业级设备池、音频链路、稳定性、性能和报告看板。

## 7. MVP 范围

### MVP 必须包含

1. **Windows 本机 ADB 真机接入与设备体检**
   - 自动发现 A2 设备。
   - 显示序列号、型号、Android 版本、分辨率、电量、网络、存储、应用状态、ADB 授权状态。
   - 支持截图、控件树/页面结构采集、日志采集、录屏开关等基础证据能力。

2. **测试报告与失败证据闭环**
   - 每次执行有报告：设备信息、环境信息、用例结果、步骤、耗时、截图、日志、失败原因、导出文件。
   - 失败时自动保留证据，不依赖人工补充。

3. **基础 UI 操作编排与执行**
   - 先覆盖：点击、下拉/滑动、输入、返回、等待、断言、截图。
   - 点击、输入、断言默认必须基于 resource-id、text、description、accessibility label、稳定层级路径等语义定位。
   - 若某个控件确实只能通过像素坐标触达，不允许放弃自动化测试；可使用像素坐标作为最后兜底，但必须在用例、执行日志和报告中显式标注“像素坐标兜底”，说明原因、风险、设备分辨率/方向和后续可测试性改造建议。
   - 第一条标杆用例：在 A2 真机上完成一个“下拉框/下拉面板/设置项”类基础操作并生成报告。

### MVP 不包含但预留方向

- 音频链路、稳定性长跑、性能趋势、多设备并发、兼容性矩阵、CI/CD、质量门禁、企业权限。

## 8. Won't Have（当前阶段明确不做）

1. 不做云端 SaaS 设备农场。
2. 不做完整低代码测试平台/AI 自动生成用例。
3. 不做大规模多租户、组织权限、审计、计费等企业 SaaS 能力。
4. 不承诺第一版覆盖全部音频硬件指标。
5. 不做固件刷写、root、底层驱动诊断等高风险设备维护能力。
6. 不做“所有测试都自动化”的虚假承诺；MVP 只验证可重复、可证据化、价值最高的核心路径。

## 10. 验收标准

### 10.1 设备接入与体检

- Given 当前 Windows 电脑已连接 A2 真机且用户已允许 USB 调试，When 用户打开平台并点击“扫描设备”，Then 平台在 60 秒内展示设备序列号、型号、Android 版本、分辨率、电量、网络、存储、ADB 状态。
- Given ADB 未授权或设备未连接，When 用户点击“扫描设备”，Then 平台显示明确失败原因和下一步处理建议。
- Given 设备已连接，When 用户点击“采集截图/日志/页面结构”，Then 平台保存对应证据并能在报告中引用。

### 10.2 基础 UI 操作

- Given A2 处于指定起始页面，When 用户运行“下拉框/下拉面板”标杆用例，Then 平台优先基于稳定语义定位完成下拉/点击/等待/断言，并记录每一步截图和结果；若某控件只能通过像素坐标触达，Then 仍需自动化执行，但报告必须显式标注“像素坐标兜底”、使用原因、坐标参数、设备分辨率/方向和稳定性风险。
- Given 页面元素未出现，When 用例等待超过配置时间，Then 用例失败并保存当前截图、页面结构、日志和失败步骤。
- Given 操作过程中设备锁屏/断连，When 平台检测到状态异常，Then 当前用例终止或标记为阻塞，并在报告中标记设备异常原因。

### 10.3 报告与证据

- Given 任意一次用例执行完成，When 用户打开报告，Then 能看到执行时间、设备信息、用例列表、通过/失败状态、失败步骤、截图、日志、录屏/附件入口。
- Given 用例失败，When 研发查看报告，Then 无需询问测试人员即可获得可复现线索：操作步骤、失败断言、当时页面、设备状态和日志片段。
- Given 项目负责人查看多次执行结果，When 进入报告列表，Then 可以按时间、设备、版本、通过率、失败模块筛选。

## 11. 风险与应对

| 风险 | 影响 | 应对方向 |
|---|---|---|
| Windows ADB 驱动/授权不稳定 | 阻断第一步接入 | MVP 优先做设备体检和故障指引 |
| A2 定制系统控件不可被标准 UI 自动化稳定识别 | UI 用例 flaky | 语义定位优先；推动 accessibility/test id 可测试性改造；必要时进入图像识别专项。若短期只能用像素坐标触达，则不能放弃自动化，必须作为“像素坐标兜底”执行，并在报告中显式说明原因、风险、坐标、分辨率/方向与后续改造建议 |
| 系统弹窗、权限弹窗、锁屏打断执行 | 用例误失败 | 边界状态识别和失败证据进入 MVP/P1 |
| 音频链路需要外部设备或专用夹具 | P2 成本和周期不确定 | 先定义可验证指标，再决定是否引入外部音频采集方案 |
| “全部测试”导致范围膨胀 | MVP 失焦 | 严格按 RICE：MVP 只做 3 件事 |
| 报告只做展示不做诊断 | 研发无法复现 | 失败证据闭环排到 MVP 第 2 |
| 多设备并发提前做 | 拖慢首个真机闭环 | 多设备放 P2/P3，先保证单设备稳定 |

## 13. 成功指标

- 首次接入：新 Windows 环境接入 A2 并完成设备体检 ≤ 30 分钟。
- 首条用例：从设备可识别到跑通下拉/点击类 UI 用例 ≤ 1 小时。
- 证据完整率：失败用例 100% 保留截图、日志、步骤、设备状态。
- 回归效率：M2 后核心冒烟回归人工时间下降 50% 以上。
- 可复现率：研发仅凭报告即可复现或定位的失败占比 ≥ 80%。

### 核心功能

- MVP：Windows 本机 ADB 真机接入与设备体检；测试报告与失败证据闭环；基础 UI 操作编排与执行。
- P0 质量约束：自动化用例必须语义定位优先；如果某功能只能靠像素坐标触达，不能放弃自动化测试，但必须作为“像素坐标兜底”显式记录在用例、日志和报告中，说明原因、坐标、设备分辨率/方向、稳定性风险和后续可测试性改造建议。
- P1：ADB/设备命令用例化；基础性能采集；稳定性/长跑测试；应用安装/启动/清数据。
- P2/P3：音频链路测试；兼容性矩阵；趋势看板；多设备并发与设备池；企业权限与质量门禁。

### 用户画像

- 主要用户：测试负责人/自动化测试工程师，需要从 0 在 Windows + A2 真机上建立可重复、可证据化的自动化回归。
- 次要用户：Android/设备研发，需要复现问题和获取日志证据；项目/产品负责人，需要交付质量看板；未来设备池管理员，需要多设备调度和资产管理。
